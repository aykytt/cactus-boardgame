00 — Origin

A physical game that couldn't ship.
Now it can.

CACTUS started as a wooden board game — a 3D spatial strategy game designed for two players. It was never a side project. In 2014, it entered the Konsept 2014 "Zeka Geliştirici Oyuncak Tasarımı" competition organized by IMMIB, and reached the finals. The design was exhibited in the official IMMIB catalogue that year.

In 2018, CACTUS was exhibited in the main hall of the 87th İzmir International Fair. Not a design fair — an international trade exhibition. The game held its own.

Then it stalled. Physical production economics didn't work. The tooling cost for a wooden board game at commercial scale was prohibitive for an independent team. 3D printing was tested as an alternative — small batches were produced, the game was playable, but the unit economics never reached a point where commercialization made sense.

The design sat. The rules didn't change. The pieces didn't change. The game was still sound. It just had no viable path to reach players.

In 2026, that path opened. A browser-based 3D implementation removes every friction point that killed the physical version — no manufacturing, no inventory, no retail. The constraint that blocked the game for over a decade is simply gone.

01 — How it was built

AI wrote the code.
Every decision was human.

For everyone

This project was built using AI as the primary implementation tool. Claude (Anthropic) generated all code — Three.js rendering, game logic, Firebase integration, mobile gesture handling, cross-browser compatibility fixes. No line of production code was written by hand.

That's the easy part to describe. The harder part: what does that actually mean for how the project was run?

It means the work shifted entirely to decisions. What should the gesture model be? What does "valid placement" mean for a 3D piece on a 3D board? When should a continuity bonus trigger — and what is the correct definition of "unbroken sequence" in a 3D space? What accessibility standard applies, and is AA sufficient or do we need AAA? Which analytics events carry signal, and which are noise?

None of those questions have answers that can be generated. They require product judgment, design experience, and domain knowledge. AI handled implementation. The team handled everything else.

02 — UX / UI

Design decisions, not design deliverables.

For UX/UI designers

The UX process here didn't produce wireframes or prototypes in the traditional sense. Every UX decision was made live, against a working build, with immediate feedback. The fidelity was always production-level.

What follows is a record of the decisions that shaped the experience — their problem context, the chosen direction, and what changed.

D-01 Platform architecture — single file, zero friction ▼
Constraint: The game needed to run in any browser, on desktop and mobile, without installation or app store friction. Installation is a conversion killer for unproven games.
Decision: Single HTML file architecture. No build tooling. No dependencies to install. Three.js loaded via CDN. Deployed on Vercel with GitHub CI/CD — every commit auto-deploys.
Outcome: Any device, any browser, shareable via a single URL. Zero onboarding friction. Playtesters could start a game within 3 seconds of receiving a link.
D-02 Mobile gesture model — separating exploration from commitment ▼
Problem: On desktop, mouse hover provides feedback before commitment — users can explore positions before placing. On mobile, every touch is simultaneously a hover and a tap. The initial implementation treated any tap as a placement trigger, which broke the exploratory flow entirely.
Decision: Separated ghost positioning from placement confirmation. One-finger drag updates the ghost in real time. Placement only fires when the user explicitly taps "Place" — and only when the ghost is green (valid). Two-finger drag rotates the 3D scene. This maps to existing platform conventions.
Outcome: Consistent with platform conventions. Users explore freely before committing. Accidental placements eliminated. Mobile experience is now equivalent to desktop, not a degraded version of it.
D-03 Safari pinch-to-zoom — documented constraint, specific fix ▼
Problem: On iOS Safari, two-finger pinch was zooming the entire browser viewport instead of the game scene. The standard user-scalable=no directive is ignored by Safari since iOS 10 — a known platform constraint with no simple workaround.
Decision: Intercept two-finger touchmove events with passive: false and call preventDefault(). Combined with maximum-scale=1.0 in the viewport meta, this prevents viewport zoom while keeping the in-game pinch-to-zoom functional at the Three.js level.
Outcome: Consistent experience across Chrome (Android), Safari (iOS), and all tested desktop browsers. No viewport jank on iOS.
D-04 Accessibility — WCAG AA/AAA as Definition of Done ▼
Constraint: A color-heavy 3D game risks failing basic readability under color vision deficiencies. The original design used a warm-to-cool gradient for both teams — visually clean, but potentially indistinguishable under deuteranopia (affects approximately 8% of males).
Decision: WCAG AA/AAA contrast ratios set as DoD criteria — not aspirational. A Material Lab tool was built separately to simulate 5 color vision conditions (Deuteranopia, Protanopia, Tritanopia, Achromatopsia, Normal) across 24 color/material combinations, enabling informed decisions before applying changes to the main build.
Outcome: Color decisions are testable and defensible. The Material Lab functions as both a design tool and a process artifact that demonstrates the decision methodology.
D-05 UI system scope — deliberate constraint ▼
Trade-off: Building a full component system and design tokens upfront would slow core game development significantly. Shipping without any standards would create inconsistency that's expensive to fix later.
Decision: Minimum viable UI system — CSS custom properties for color tokens, consistent spacing rhythm, WCAG-compliant contrast. No component library. UI decisions were explicitly not allowed to block gameplay progress. The debt was accepted and documented.
Outcome: Core game shipped in days, not weeks. UI debt is backlogged. The constraint proved the principle: accepting known debt explicitly is more productive than trying to eliminate all debt before shipping.
D-06 Multiplayer architecture — online-first removes the physical constraint ▼
Constraint: The physical game required both players to be in the same room. This is the single largest limitation of any physical board game — and the reason CACTUS never scaled beyond demo contexts.
Decision: Firebase Realtime Database for online multiplayer. Room code system (4-letter, 2-hour TTL). Real-time move sync across both clients. Offline mode retained for same-screen play. Lobby handles mode selection, player naming, countdown, and sync initialization.
Outcome: The constraint that blocked the game for a decade is eliminated. Two players anywhere can play a full match. This is also the primary product unlock — it changes who the game can reach.

The 3D board is a 2×2×8 grid of cubes — 32 total. Pieces attach to outward-facing surfaces. Placement validity is computed by raycasting from the mouse/touch position onto the active placement plane, snapping to the nearest valid grid offset, and running collision detection against both board cubes and previously placed pieces.

// Board geometry: x ∈ {0,1}, y ∈ {0..7}, z ∈ {0,1} // Valid placement: attached to board surface, no collision with placed pieces // Ghost system: real-time preview (green=valid, red=invalid) // Rotation: 24 unique orientations per piece (via quaternion deduplication) // Move notation: P{id}·R{rotIdx}·({gx},{gy},{gz})·{face}

Piece rotation generates all 24 orientations via quaternion application, then deduplicates by coordinate fingerprint. This avoids redundant rotations for symmetric pieces and keeps the valid move set precise.

The continuity bonus system tracks threshold state per player across the game session — not per-move. Once the 8-unit threshold is crossed, that bonus cannot trigger again for that player. The 16-unit threshold is tracked independently. This prevents re-triggering on subsequent moves that maintain but don't extend the sequence.

Firebase security rules restrict write access: rooms can be created when the path doesn't exist, joined when status is 'waiting', and game-over events can only be written when a game is in 'playing' state. This prevents room hijacking and stale writes.

// Firebase Realtime DB rules (production) rooms/$roomId: .read: true .write: !data.exists() || status === 'waiting' || status === 'playing' gameOver/.write: parent.status === 'playing'
04 — Product & Analytics

What we're measuring and why.

For product owners / decision makers

Standard page-view analytics are meaningless for a single-page game. A session that starts and ends on the same URL could be a 30-second bounce or a 25-minute match. The distinction matters entirely.

The instrumentation strategy here treats the game as a stateful session with discrete events. Every meaningful state transition is tracked with its full context.

game_start
Fired when the game engine initializes after lobby. Captures mode, platform, and session context.
platform · game_mode · replay_number
piece_placed
Fired on every move. Full notation string — piece identity, rotation, board coordinates, attachment face.
player · piece_name · piece_units · move_number · notation · platform
game_end
Fired at game conclusion. Raw scores, penalty deductions, win type, full session metrics.
winner · win_type · score_raw · score_final · penalty · session_seconds · game_notation
game_abandoned
Fired on beforeunload if the game is active. Captures drop-off point and session length at abandonment.
move_number · scores · session_seconds · platform
inspect_entered
Post-game board inspection mode. Signals engagement with the result beyond just seeing the score.
total_moves · platform
game_replay
New game started without leaving. Strongest retention signal — player chose to play again immediately.
replay_number · platform
The notation system enables sequence analysis. Every move is recorded as a structured string: piece ID, rotation index, board coordinates, attachment face. The full game is stored as an ordered sequence. This makes post-game analysis possible at the move level — not just session level. Which pieces are never played. At which move number do most games end. How often does the Lock Win condition occur relative to Score Win.

The multiplayer decision was also a product decision. The physical game required both players in the same room — a constraint that limited every demo, every test, every potential player. Online multiplayer removes that constraint entirely. It changes the distribution model: the game can now be shared as a link, played asynchronously across cities, and tested with people who have never seen a physical copy.

Penalty scoring was added to close a game theory gap: without end-of-game penalties for unused pieces, a player could accumulate points conservatively while hoarding high-value pieces. The penalty — each unused piece deducted by its unit count — forces a trade-off between tactical caution and piece utilization. The win screen displays both raw and final scores with a breakdown, so the penalty's effect on the match outcome is visible.

05 — Game Complexity

How many different games of CACTUS can be played?

For anyone who wants the deep version

Before this question can be answered, the game needs a notation system — a language for describing positions precisely enough to count them.

Every surface on the board is defined by two coordinates: which cube (x, y, z) and which face L/R/F/B/T/D. A complete move records piece identity, rotation index, attachment coordinates, and face direction.

Move Notation — Format
P2 · R4 · (0,0,0) · F
4-unit L piece (P2) · 5th unique rotation (R4) · attached at cube (0,0,0) · front face

The board has 32 cubes in a 2×2×8 grid, with 72 outward-facing surfaces as potential attachment points. 24 of these belong to the start zones (y=0 and y=7). The first move must contact a start zone surface.

Across all 10 pieces and all valid orientations, there are 1,935 valid first-move options. This is the computed value — not an estimate.

Game tree size is estimated at approximately 10³⁸. For reference:

Tic-tac-toe
~10⁵
Checkers
~10³¹
CACTUS
~10³⁸
Chess
~10¹²³

CACTUS sits between Checkers and Chess in game tree complexity. This is a consequence of three-dimensional placement, 24 rotation states per piece, and the absence of piece repetition — each piece is placed exactly once per player.

The full analysis — board geometry, orientation counting, valid move computation, and game tree estimation — is documented in the Complexity Report.

The game is live and in a closed playtest phase. Analytics are running. The focus is on validating game mechanics, cross-device behavior, and interaction clarity before broader distribution.

What's working: offline same-screen play, online multiplayer via room codes, cross-platform gesture handling, real-time scoring with penalty calculation, and the full analytics pipeline. Firebase security rules are in production configuration.

What's in progress: online multiplayer sync edge cases, move replay system architecture, and broader playtesting.

On the roadmap

A move replay system — after a game ends, both players can step through each move from the beginning, viewing the exact board state at any point. No undo, no branching — pure post-game review. When the final move is reached, the game becomes playable again. This turns every match into a learning artifact, and enables the kind of session analysis that makes the notation system meaningful.
